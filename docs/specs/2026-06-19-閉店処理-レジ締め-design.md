# 設計書：閉店処理（レジ締め）の内製化

## 1. 背景と目的

hpb-calendar の 1cs POS 内製化の第3サブプロジェクト。
フェーズ1（予約管理）・フェーズ2（会計＝レジ・金種・伝票）を内製化済み。会計フェーズで明示的に次フェーズへ先送りした「レジ金の金種別在高カウント・レジ締め」を本フェーズで内製化する。

- 会計確定で `sales-log.json` に現金/カード/QRの売上が追記される。本フェーズはその**現金売上**を軸に、閉店時のレジ金を金種別に数え、理論在高との過不足を確認し、翌日繰越分を残して入金額を確定する**日次締め**を実装する。
- 「集計」は既存の `report.html` ＋ `api/sales.js` で充足済みのため本フェーズの対象外。

### 決定事項（brainstorming 2026-06-19）

| 項目 | 決定 |
|---|---|
| スコープ | 閉店処理（レジ締め）に集中。集計は既存 report.html で充足済み |
| 釣り銭準備金 | **固定繰越**（毎日同じ金額を残す）。締め画面で調整可・既定値は前回の締めから復元 |
| 営業中の入出金 | **記録しない**（シンプル運用）。理論在高＝釣り銭準備金＋当日現金売上のみ |
| 確定後の扱い | **何度でもやり直せる（日付キーで upsert・ロックなし）** |
| 過不足判定 | **現金のみ**。カード/QRは参考表示 |

## 2. ゴール / 非ゴール

**ゴール**
- 独立ページ `close.html`「🧾 レジ締め」を実装する（既存の customers.html / report.html と同じく機能＝独立ページの構成。admin の topnav に「レジ締め」リンクを追加）
- 当日の現金売上を `sales-log.json` から自動集計する
- 金種別（¥10,000〜¥1）の枚数入力で実在高をライブ計算する
- 理論在高（釣り銭準備金＋現金売上）・過不足（実在高−理論在高）・入金額（実在高−釣り銭準備金）を算出する
- 締めを `daily-close.json` に日付キーで upsert（何度でもやり直し可）する
- 画面ロード時に既存の締め記録があれば金種・準備金・メモを復元する
- カード/QR売上を参考表示する

**非ゴール（別サブプロジェクト or 対象外）**
- 営業中の現金入出金（釣銭補充・経費現金払い）の記録
- レジ締め確定によるその日のロック（会計の計上漏れ防止）
- 集計ダッシュボードの拡充（既存 report.html で充足）
- 金種別の翌日繰越内訳（繰越は合計額のみ・金種内訳は持たない）
- 締めの承認フロー・複数スタッフ権限
- 月次/期間の締め（本フェーズは日次のみ）

## 3. データモデル

### 3.1 締め台帳 `daily-close.json`（R2／新規）
日付キーで upsert する配列。1日1レコード。
```json
[{
  "date": "2026-06-19",
  "float": 30000,
  "cashSales": 12800,
  "expectedCash": 42800,
  "denominations": { "10000": 4, "5000": 0, "2000": 0, "1000": 2, "500": 1, "100": 3, "50": 0, "10": 0, "5": 0, "1": 0 },
  "countedCash": 42800,
  "overShort": 0,
  "deposit": 12800,
  "cardSales": 8000,
  "qrSales": 0,
  "note": "",
  "closedAt": "2026-06-19T11:00:00.000Z"
}]
```

| フィールド | 意味 | 算出 |
|---|---|---|
| `date` | 締め対象日（YYYY-MM-DD・upsertキー） | 入力 |
| `float` | 釣り銭準備金（固定繰越） | 入力（既定＝前回締めの float、無ければ既定定数） |
| `cashSales` | 当日の現金売上 | sales-log から自動集計（`payment==="cash"` かつ当日） |
| `expectedCash` | 理論在高 | `float + cashSales` |
| `denominations` | 金種別枚数 | 入力 |
| `countedCash` | 実在高 | `Σ(額面 × 枚数)` |
| `overShort` | 過不足 | `countedCash - expectedCash`（+過剰 / −不足） |
| `deposit` | 入金額（繰越除く） | `countedCash - float`（負なら0クランプ） |
| `cardSales` / `qrSales` | 参考売上 | sales-log から自動集計（当日・参考表示のみ） |
| `note` | メモ | 入力（任意） |
| `closedAt` | 締め確定時刻 | サーバー時刻（ISO） |

**金種定義（額面の降順）**：`[10000, 5000, 2000, 1000, 500, 100, 50, 10, 5, 1]`

### 3.2 釣り銭準備金の既定値
- 固定繰越のため、`GET /api/close` は「前回（最新日付）の締めレコードの `float`」を既定値として返す。
- 締め記録が一件も無い場合の初期既定は定数 `DEFAULT_FLOAT = 30000`（`lib/close.js` に定義）。
- ユーザーが締め画面で調整した `float` は当該日のレコードに保存され、以降の既定値になる。

## 4. 計算コア `lib/close.js`（純粋関数・新規）

```js
const DENOMS = [10000, 5000, 2000, 1000, 500, 100, 50, 10, 5, 1];
const DEFAULT_FLOAT = 30000;

// 金種別枚数 → 実在高
function countCash(denominations) { /* Σ(額面×枚数)・不正値は0扱い */ }

// 締めの確定計算（純粋）
// 入力: { float, cashSales, denominations }
// 出力: { countedCash, expectedCash, overShort, deposit }
function computeClose({ float, cashSales, denominations }) {
  const countedCash  = countCash(denominations);
  const expectedCash = (Number(float) || 0) + (Number(cashSales) || 0);
  const overShort    = countedCash - expectedCash;
  const deposit      = Math.max(0, countedCash - (Number(float) || 0));
  return { countedCash, expectedCash, overShort, deposit };
}

module.exports = { DENOMS, DEFAULT_FLOAT, countCash, computeClose };
```

**境界値**：枚数が負/NaN/未指定→0扱い。`float` 負→0扱い。`deposit` は負を0にクランプ（実在高が準備金未満でも入金額は0）。`overShort` は符号そのまま（不足はマイナス）。

## 5. API `api/close.js`（新規）

会計フェーズの `api/sales.js` / `api/checkout.js` と同じ認証・CORS パターンを踏襲（`Authorization: Bearer ${COMINGSOON_PASSWORD}`）。

### 5.1 `GET /api/close?date=YYYY-MM-DD`
当日の締め画面に必要な値を返す。
```json
{
  "date": "2026-06-19",
  "existing": { /* daily-close.json の当日レコード or null */ },
  "cashSales": 12800,
  "cardSales": 8000,
  "qrSales": 0,
  "defaultFloat": 30000
}
```
- `cashSales`/`cardSales`/`qrSales`：`sales-log.json` を当日でフィルタし `payment` 別に `total`（無ければ旧 `finalPrice`）を合計。
- `defaultFloat`：当日レコードがあればその `float`、無ければ「最新日付の締めレコードの `float`」、それも無ければ `DEFAULT_FLOAT`。
- `existing` があれば金種・メモを画面に復元する。

### 5.2 `POST /api/close`
締めを確定（upsert）する。
- 入力：`{ date, float, denominations, note }`
- サーバーで `cashSales`（当日現金売上）を再集計し、`computeClose` で `countedCash/expectedCash/overShort/deposit` を算出（クライアント計算は信用しない）。
- `cardSales/qrSales` も再集計して保存。
- `daily-close.json` を読み、同 `date` があれば置換、無ければ追加（upsert）。`closedAt` を更新。
- 返却：保存したレコード全体。

## 6. UI（独立ページ `close.html`・「🧾 レジ締め」）

admin の topnav タブは各機能＝独立ページへのリンク（顧客→customers.html / レポート→report.html）。レジ締めも同パターンで `close.html` を新規作成し、`report.html` のログイン/認証（`token` + `Bearer`）構造をミラーする。admin.html・report.html の topnav に「🧾 レジ締め」リンク（`/close`）を1つ追加する。

```
🧾 レジ締め            2026年6月19日(木)  [◀前日][今日][翌日▶]

  釣り銭準備金   [ ¥30,000 ]   ← 固定・編集可（既定=前回締め）
  当日現金売上    ¥12,800       ← 自動（GET /api/close）
  ─────────────────────────
  理論在高        ¥42,800       （準備金＋現金売上）

  金種別カウント（実際の現金を数える）
   ¥10,000 [4] = ¥40,000
   ¥5,000  [0] = ¥0
   ¥2,000  [0] = ¥0
   ¥1,000  [2] = ¥2,000
   ¥500    [1] = ¥500
   ¥100    [3] = ¥300
   ¥50 [0] / ¥10 [0] / ¥5 [0] / ¥1 [0]
   実在高          ¥42,800

  過不足           ±0           （緑=0 / 赤=不足 / 青=過剰）
  入金額（繰越除く） ¥12,800
  メモ [____________]

  参考：カード ¥8,000 / QR ¥0
                              [ 締め確定 ]
```

- 金種の枚数入力（`oninput`）で実在高・過不足・入金額をライブ再計算（クライアント側 `lib/close.js` 相当のロジックをJSで実装）。
- 過不足の色：0=緑 / 負（不足）=赤 / 正（過剰）=青。
- 日付ナビ（前日/今日/翌日）で `GET /api/close?date=` を再取得し、既存締めを復元。
- 「締め確定」で `POST /api/close` → 成功トースト＋保存値で再描画。

## 8. 後方互換・既存への影響

- 既存ファイルへの破壊的変更なし。`lib/close.js`・`api/close.js`・`close.html` 新規。既存 `admin.html`/`report.html` の topnav に「レジ締め」リンクを1つ追加するのみ。
- `sales-log.json` の読み取りは `api/sales.js` と同じく `total`→旧`finalPrice` フォールバックを踏襲。
- `server.js` は `api/` を動的マウントするため `api/close.js` 追加で自動的に `/api/close` が有効になる。`/close` ページのクリーンURLは `server.js` に `app.get('/close', ...)` を、`vercel.json` に `{ "src": "/close", "dest": "/close.html" }` を追記（`/api/(.*)` ルートより前ではなく既存ページルート群に並べる）。
