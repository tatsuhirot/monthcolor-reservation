# 設計書：予約管理の内製化（1cs相当の予約登録・自前DB保存）

## 1. 背景と目的

MONTH COLOR の予約・会計は現在 **1cs.jp の「coming-soon POS」（register-app）** で運用している。
本プロジェクト（hpb-calendar）は既にそのPOSの大部分を内製化してきた：

| 1cs POSモジュール | 自前システムの現状 |
|---|---|
| 📅 予約管理 | △ admin.html の「＋新規予約」モーダル + `api/reserve.js`（基本項目のみ） |
| 📊 集計管理 | ◯ `api/sales.js` |
| 🧾 伝票作成 | ◯ `api/receipt.js` / `api/checkout.js` |
| 🔍 顧客検索 | ◯ `api/customers.js`（sales-log由来） |
| ⚙️ 設定 | △ 部分的 |
| 🔒 閉店処理 | ✕ なし |

最後に1csへ依存している中核「**予約管理（予約登録）**」を内製化し、**予約データは自前ストレージ（Cloudflare R2）を正**とする。
SalonBoard / 1cs への送信は当面切り離す（依存ゼロを目指す）。

> 補足：SalonBoard 連携は Akamai の多層防御（自動ログイン不可・連続アクセスでスロットリング・1アカウント1セッションでスタッフの朝ログインに蹴られる）により安定運用が困難。自前を正とすることでこの問題から独立する。

## 2. ゴール / 非ゴール

**ゴール（本サブプロジェクト）**
- 1cs相当の予約「登録」フォームを admin 内に実装する
- 1cs相当の予約「一覧（予約プレート）」を admin に表示する
- 予約データを自前R2に保存し、これを正とする
- メニューを単価付きマスタ（CSV由来）から選択できるようにする

**非ゴール（別サブプロジェクトで実施）**
- 会計（レジ・金種・伝票）… `checkout.js`/`receipt.js` を活かして次フェーズ
- 集計管理 … `sales.js` を活かして次フェーズ
- 閉店処理（未実装）
- 個別スタッフ指名マスタ（MONTH COLOR は個人指名運用なし＝**指名はフラグのみ**で確定）
- ネット予約（HPB/SalonBoard）の自動取込（経路フラグ `net` は持つが取込は別途）

## 3. 参照した一次情報

### 3.1 1cs の予約データモデル（予約プレート `app-reserve-plate-view` より）
予約プレート1枚が持つ項目：
- 顧客名 ＋ **カナ（フリガナ）**
- 状態：**予約 → 来店 → 未精算 → 精算済み**
- 予約時刻 / **指名フラグ**（※全件「美容師の指名はできません」＝個人指名なし）
- サービス区分（例：ヘアカラー）
- メニュー詳細（例：根元染め(リタッチ)、カラー＋トリートメント、`＆` で複数結合：根元染め(リタッチ)＆炭酸泉）
- 伝票No.・数量（会計連携）
- 予約経路（`net-reservation` クラス＝ネット予約フラグ）

### 3.2 メニューマスタ（`items_MONTH COLOR_20260615.csv`／Shift-JIS(CP932)）
列：`技術コード, 技術分類, 技術名, 税込技術料, 税抜技術料, 基本販売数量, レジ表示`
**63件 / 14分類**。分類例とサービス区分対応：

| 技術分類（例） | サービス区分 |
|---|---|
| 根元染め(リタッチ) / 全体染め(ショート〜ダブルロング) / トリートメント / 【新規】カラー＋… / その他ヘアカラー / 補正 | hair |
| セルフホワイトニング | white |
| まつ毛パーマ | lash |
| ドライヘッドスパ | spa |
| 割引 | （調整・カテゴリ非依存） |

> CSV に**所要時間（分）が無い**ため、所要時間は `api/_shared.js` の `MENU_DURATION` の考え方で**分類→所要時間**を補完する（例：リタッチ70分）。料金は CSV を正とする。

## 4. データモデル

### 4.1 メニューマスタ `menu-master.json`（R2／新規）
CSV を変換して生成。配列要素：
```json
{
  "code": "T010001",
  "category": "根元染め(リタッチ)",
  "name": "【平日通常】リタッチ",
  "priceIncTax": 2980,
  "priceExTax": 2710,
  "defaultQty": 1,
  "service": "hair",          // 分類→hair/white/lash/spa を付与
  "durationMin": 70           // 分類→MENU_DURATION で補完
}
```
- 変換スクリプト：`scripts/import_menu_master.js`（CSV CP932 → JSON、`service`/`durationMin` を付与）
- 配信API：`GET /api/menu`（フォームのメニュー選択で使用）

### 4.2 予約レコード（`reservations-queue.json` を拡張）
既存：`{ id, type, status, data:{date,time,name,menuName,menuId,staffCategory,phone,email,memo}, createdAt }`
拡張後の `data`：
```json
{
  "date": "2026-07-02",
  "time": "12:30",
  "name": "山田 花子",
  "nameKana": "やまだ はなこ",      // 追加
  "customerId": "phone:09012345678", // 追加（カルテ紐付け。電話キー）
  "phone": "090-1234-5678",
  "email": "",
  "menuItems": [                      // 追加（複数＆対応・単価付き）
    { "code": "T010001", "name": "【平日通常】リタッチ", "price": 2980, "durationMin": 70 }
  ],
  "service": "hair",                  // menuItems から主区分を決定（旧 staffCategory 相当）
  "source": "phone",                 // 追加: net | phone | walkin | manual
  "nominated": false,                 // 追加: 指名フラグのみ
  "memo": "電話予約",
  "slipNo": null                      // 会計時に付与（次フェーズ）
}
```
- 旧 `menuName`/`menuId`/`staffCategory` は後方互換のため当面残す（`service` は `staffCategory` の役割を引き継ぐ）。
- **状態ライフサイクル**：`status` を予約管理用に整理
  `pending(予約)` → `arrived(来店)` → `unpaid(未精算)` → `paid(精算済)`／`cancelled`
  既存の `processing/completed/failed` は SalonBoard 送信用だったため、送信OFF時は使わない。
- **占有スロット計算**：`menuItems` の合計所要時間で `getOccupiedSlots` を算出（最長 or 合計＝要実装時確認、既定は**合計**）。

## 5. UI / フォーム（admin 内）

### 5.1 予約登録フォーム（1cs相当）
既存「＋新規予約」モーダル（admin.html）を置き換え／拡張：
1. **顧客検索**：電話／名前／カナで `GET /api/customers` を検索 → ヒットで氏名・カナ・電話を自動入力。新規はそのまま手入力。
2. **メニュー選択**：分類タブ → メニュー一覧（`/api/menu`、単価表示）→ 複数追加可（`＆`）。合計金額・合計所要時間を表示。
3. **日時・枠選択**：既存 `slot-grid` を流用。`menuItems` 合計所要時間と `CAPACITY`（hair6/white1/lash2/spa1）で空き判定。
4. **連絡先・予約経路・備考**：phone/email/source/memo。
5. **登録**：`POST /api/reserve` を拡張し、自前R2へ保存（SalonBoard送信はOFF）。確認メールは email がある時のみ（既存挙動踏襲）。

### 5.2 予約一覧（プレート）
admin の予約管理タブを 1cs の予約プレート相当に整理：
- 1行＝1予約：時刻 / 顧客名（カナ） / サービス区分 / メニュー / 状態バッジ / 経路アイコン / 指名
- 状態で色分け（予約 / 来店 / 未精算 / 精算済）

## 6. アーキテクチャ（コンポーネント）

| 単位 | 役割 | 依存 |
|---|---|---|
| `scripts/import_menu_master.js` | CSV→`menu-master.json` 変換（service/duration付与） | CSV, lib/storage |
| `GET /api/menu` | メニューマスタ配信 | menu-master.json |
| `POST /api/reserve`（拡張） | 予約をR2に登録（自前が正・送信OFF） | _shared, storage, resend(任意) |
| `api/_shared.js`（拡張） | menuItems合計の所要時間/空き枠計算、状態定義、分類マップ | storage |
| admin.html（拡張） | 予約登録フォーム＋プレート一覧 | /api/menu, /api/reserve, /api/customers |

各単位は単一責務・明確なI/Oを持ち、独立してテスト可能にする。

## 7. データフロー

```
[スタッフ] admin予約登録フォーム
   → 顧客検索 GET /api/customers
   → メニュー GET /api/menu
   → 登録 POST /api/reserve
        → _shared: 空き枠チェック（menuItems合計所要時間・CAPACITY）
        → storage.put reservations-queue.json（status=pending=予約）
        → (email有) 確認メール
   → admin プレート一覧 GET /api/reservations に反映
```

## 8. エラーハンドリング
- 必須（日付・時刻・氏名・メニュー1件以上）未入力 → 400
- 枠満席 → 409（既存の満席チェック踏襲、menuItems合計で再計算）
- メニューマスタ未ロード → /api/menu は空配列を返し、フォームは「メニュー未登録」を表示
- R2書込失敗 → 500、フォームはトーストでエラー表示しデータ保持

## 9. テスト方針
- `_shared` の所要時間/空き枠計算：単体テスト（複数menuItems・区分別CAPACITY・日跨ぎ無し）
- `import_menu_master.js`：CSV(CP932)→JSON 変換のスナップショット（63件・14分類・service/duration付与）
- `POST /api/reserve`：必須欠落=400 / 満席=409 / 正常=保存 のケース
- admin フォーム：顧客検索自動入力・メニュー複数追加・合計表示の手動E2E

## 10. 実装フェーズ（writing-plans で詳細化）
1. メニューマスタ：`import_menu_master.js` + `menu-master.json` + `GET /api/menu`
2. データモデル拡張：`_shared.js`（menuItems所要時間・状態・分類マップ）＋ `reserve.js` 拡張
3. 予約登録フォーム：admin.html（顧客検索・メニュー選択・枠選択・合計）
4. 予約プレート一覧：admin.html（状態・経路・指名の表示整理）
5. 結線・手動E2E・既存予約データの後方互換確認
